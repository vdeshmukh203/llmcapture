01Overview

A Chromium browser extension that produces cryptographically chained, tamper-evident forensic logs of AI chat interactions across ChatGPT, Claude, and Gemini. Designed for scenarios requiring verifiable, court-admissible records of AI session content — including academic research, legal proceedings, and compliance audits.

Design intent

Produce session logs where any post-capture modification — insertion, deletion, reordering, or content alteration — is cryptographically detectable via SHA-256 hash chaining. Each entry's chain hash commits to all prior entries, the entry's own content, its position index, and its timestamp. No silent corruption is possible.

Threat model

Adversary with access to the session JSON log files but not the extension runtime. Can modify JSON fields, delete entries, reorder records. Cannot replay live SHA-256 computation without access to the chain seed. Cross-session hash transplantation is defeated by the session-unique chain seed.

Platform scope

ChatGPT — chat.openai.com Claude — claude.ai Gemini — gemini.google.com Extensible via new extractor modules

Architecture summary

Three-tier runtime: a MV3 service worker (background.js) holds authoritative chain state; a content script (content.js) injects platform-specific extractors and relays captured entries; an extension popup provides session control. Persistence via the File System Access API writes JSON session logs to a user-designated directory. An offline React verifier (chain_verifier.tsx) replays hash chains for integrity validation without requiring the extension.

03Data pipeline

Nine distinct stages from DOM mutation event to persisted, cryptographically chained log entry.

1
DOM mutation event
MutationObserver in the site-specific extractor fires on new assistant message node insertion. Target selector is platform-specific — e.g. [data-message-author-role="assistant"] for ChatGPT. Streaming state detected via presence of a streaming cursor class or loading spinner attribute.
2
Streaming fence
Extractor checks for streaming-in-progress marker. If streaming is active, schedules a re-check via debounced timeout — avoids capturing partial content mid-stream. Commit only proceeds when streaming marker is absent and content has been stable for the platform-tuned debounce window.
3
Content extraction
Extracts raw text via innerText on the settled content node. Strips extractor-specific prefix contamination (bug #4 fix). For Gemini: uses integer positional cursor rather than content-hash cursor to avoid phantom recommits after post-streaming DOM reformatting (critical bug #1 fix).
4
Deduplication check
Compares extracted content against the last committed entry's content hash. If matched, DUPLICATE status is assigned — but the entry is still committed and chained (failure-as-data principle). Nothing is silently dropped.
5
Content hash computation
lib/crypto.js computes SHA-256(UTF-8(content)) via SubtleCrypto.digest(), returning a lowercase hex string. This is the per-entry content fingerprint — independent of position in chain.
6
Chain hash computation
chain_hash = SHA-256(prev_chain_hash + content_hash + entry_index + timestamp_iso). Genesis entry (Run 1, index 0) uses null as prev and records the session chain_seed. Each subsequent entry commits to the full accumulated prior state.
7
Entry assembly
Complete entry object constructed: {entry_index, session_id, run_id, role, content, content_hash, chain_hash, prev_chain_hash, timestamp_iso, status, platform, metadata}. Status is one of OK | INVALID | ERROR | DUPLICATE | EMPTY_RESPONSE.
8
Background relay
content.js sends entry to background.js via persistent port (chrome.runtime.connect). Background validates session state is CAPTURING before accepting. Holds authoritative chain state across all tabs — prevents race conditions on multi-tab sessions.
9
Storage write
lib/storage.js appends entry to in-memory session array. On session close, periodic flush (60s), or PAUSE, writes full session JSON to File System Access API handle. Atomic write: serialize to temp file, then rename to final path — prevents partial-write corruption.

04Cryptographic chain architecture

Hash primitive

All hashes: SubtleCrypto.digest('SHA-256', data) → ArrayBuffer → lowercase hex string. Key-less — no HMAC. Chain integrity derives from sequential commitment, not a shared secret. Verifiable by any party with access to the session JSON and a browser.

Two-hash design

content_hash — SHA-256 of entry content. Detects content tampering independently of chain position. Can be checked entry-by-entry without replaying the full chain.

chain_hash — commits to position, timestamp, and all prior state. Detects insertion, deletion, and reordering even when content_hash values are preserved.

Genesis entry — Run 1 anchoring

// Entry index 0, run_id = "run_1" — establishes chain root
{
  entry_index:     0,
  run_id:          "run_1",
  prev_chain_hash: null,              // no predecessor
  chain_seed:      "<random 32-byte hex>",  // session-unique entropy
  chain_hash:      SHA256("null" + content_hash + "0" + timestamp_iso),
  status:          "OK"
}

Chain seed generated via crypto.getRandomValues(new Uint8Array(32)). Binds the chain to a specific session instance — prevents cross-session hash transplantation attacks where a valid chain from session A is grafted into session B.

Chain propagation formula

// Each entry n (n > 0)
chain_hash[n] = SHA256(
  chain_hash[n-1]     // binds to all prior state
  + content_hash[n]   // binds to this entry's content
  + String(n)          // binds to absolute position
  + timestamp_iso[n]  // binds to capture time
)

Multi-run continuity

On PAUSE + RESUME, a new run_id is issued (run_2, run_3…). The chain continues: the first entry of run_2 uses the final chain_hash of run_1 as its prev_chain_hash. Cross-run integrity is fully preserved. A session abort (navigation, crash) is logged as an ERROR entry preserving index continuity.

Platform-tuned debounce windows: ChatGPT 300 ms · Claude 500 ms · Gemini 800 ms (extended to cover full post-streaming DOM reformat cycle).

scanLock mutex — gemini.js (critical bug #1 fix)

let scanLock = false;
let scanPending = false;

async function scanWithLock() {
  if (scanLock) { scanPending = true; return; }  // defer, don't stack
  scanLock = true;
  try {
    await doScan();
  } finally {
    scanLock = false;
    if (scanPending) {           // drain exactly one deferred scan
      scanPending = false;
      scanWithLock();
    }
  }
}

Prevents concurrent scans triggered by Gemini's post-streaming DOM reformatting from racing, producing phantom recommits, and inflating the entry count with DUPLICATE records.

lib/input-capture.js — user prompt capture

Intercepts user input before submission via two parallel hooks: (1) keydown listener on the platform textarea captures content on Enter / submit keypress. Captured user prompts flow through the identical hash chain pipeline as assistant messages, with role: "user".

07Storage layer & session log schema

Three-tier storage

In-memory (runtime): background.js service worker holds live session array. Survives tab reload via chrome.storage.session (ephemeral). Lost on browser close — periodic flushes prevent data loss.

chrome.storage.local: Lightweight state — current session_id, last chain_hash, run_id. Not used for full log storage (5 MB limit insufficient for long sessions).

File System Access API: Primary log store. User grants directory access once (persisted via StorageFoundation). Writes session_logs/ai-capture-{platform}-{timestamp}.json.

Write strategy

Each entry is appended to the in-memory array immediately upon commit. Flush triggers: (a) periodic — every 60 seconds, (b) on PAUSE, (c) full write on STOP. Atomic write: serialize full session to a temp file, then rename to the final path. Prevents partial-write corruption on crash or power loss.

File naming convention: ai-capture-chatgpt-2024-11-15T142301.json. Timestamp is session start time in local ISO format. Platform slug: chatgpt | claude | gemini.

Persistent port — entry relay

For sessions with high entry frequency, content.js opens a chrome.runtime.connect({name:'entry-relay'}) port at EXTRACTOR_READY. All CAPTURE_ENTRY messages flow through this port. Port disconnect (tab close, navigation crash) is the primary mechanism for background to detect session termination — triggers flush + SESSION_CLOSED sentinel. Eliminates per-message connection overhead and makes crash detection reliable.

09Chain verifier

Design & portability

chain_verifier.tsx compiles to a static React bundle. No backend. Accepts session JSON via file-input drag-drop. Runs entirely in-browser via SubtleCrypto. The verifier bundle + session JSON can be sent to opposing counsel or a tribunal expert who can independently verify chain integrity without installing the extension — a critical property for legal admissibility.

Verification algorithm

async function verifySession(session: Session): Promise<VerifyResult[]> {
  let prevHash: string | null = null;
  const results: VerifyResult[] = [];

  for (const entry of session.entries) {
    // 1. Verify content hash (shared utility — same as writer)
    const computedContent = entry.status === "EMPTY_RESPONSE"
      ? null
      : await sha256(entry.content);
    const contentOk = computedContent === entry.content_hash;

    // 2. Verify chain hash (same computeChainHash() as writer — critical fix)
    const computedChain = await computeChainHash(
      prevHash, entry.content_hash, entry.entry_index, entry.timestamp_iso
    );
    const chainOk = computedChain === entry.chain_hash;

    results.push({
      entry_index: entry.entry_index,
      contentOk, chainOk,
      verdict: contentOk && chainOk ? "PASS" : "FAIL"
    });
    prevHash = entry.chain_hash;
  }
  return results;
}

1013-bug audit (v10 → v11.0.0)

Severity: Critical = integrity failure / false tamper-clean signal · High = data loss or chain break · Medium = functional defect · Low = edge case

Critical — integrity failures (3)

#1 Gemini tail_cursor_signature hash cursorPost-streaming DOM reformatting invalidated stored content hashes used as positional cursors → full array rescan on each mutation → phantom recommit of prior entries → inflated entry count and chain hash corruption. Fix: switch to integer positional index cursor; add scanLock mutex; add debounced scanning with 800 ms window.
#2 Verifier / writer hash path divergencechain_verifier.tsx used a different string concatenation order than the writer (writer: prev+content+index+ts; verifier: prev+ts+content+index) → verifier reported PASS on tampered logs — a false tamper-clean signal. Fix: shared computeChainHash() utility function imported by both writer and verifier.
#3 False PASS on truncated sessionsSESSION_CLOSED sentinel absent (crash/abort) caused verifier to exit chain loop early, reporting PASS without validating tail entries. Fix: verifier now iterates all entries[] regardless of sentinel presence; emits WARN when sentinel absent.

High — data loss / chain break (4)

#4 Extractor prefix contaminationLanguage identifier token from code blocks ("javascript const x = 1") prepended to extracted content → content_hash committed against contaminated string → hash diverges from actual response content. Fix: strip leading language identifier token from code block innerText before hashing.
#5 Overbroad Gemini fallback selectorGeneric div p fallback selector captured sidebar UI chrome (navigation items, button labels) as assistant message content. Fix: selector scoped strictly to response container; generic fallback removed entirely.
#6 Buffer not flushed on PAUSEIn-memory session buffer not flushed when transitioning CAPTURING → PAUSED → entries lost if session not subsequently resumed. Fix: explicit flush call added in SESSION_PAUSE handler before observer suspension.
#7 EMPTY_RESPONSE silently droppedZero-length content after trim caused early return before chain commit → gap in entry_index sequence → chain verification FAIL on a legitimate session. Fix: EMPTY_RESPONSE status entry committed with null content_hash; chain_hash still computed and chain continuity preserved.

Medium — functional defects (4)

#8 run_id not incremented on RESUMEAll resumed entries written with run_id="run_1" → loss of pause/resume segmentation in logs; impossible to distinguish continuous from interrupted sessions. Fix: run counter incremented atomically in SESSION_RESUME handler before first entry commit.
#9 Genesis entry timestamp driftGenesis entry timestamp set at session_start but re-used as chain input after the user granted directory access (variable delay of seconds to minutes) → timestamp committed into chain_hash predates actual session data. Fix: chain_seed generated at START; genesis entry timestamp set at first actual message capture.
#10 Content.js re-injection on SPA navigationNavigation within ChatGPT (new conversation) re-injected content.js without tearing down existing MutationObserver → two active observers → duplicate entry commits with DUPLICATE status noise inflating the log. Fix: URL_CHANGE detection sends OBSERVER_SUSPEND, tears down existing observer, re-initialises cleanly.
#11 Claude artifact content in main chainArtifact panel DOM text included in main entry innerText → content_hash changed on artifact expand/collapse user interactions post-capture → hash instability. Fix: artifact containers explicitly excluded from innerText extraction; flagged in metadata.has_artifact.

Low — edge cases (2)

#12 Popup status not reflecting PAUSED stateBackground returned "CAPTURING" during async flush window on PAUSE — popup showed wrong state for several seconds. Fix: state transitioned to PAUSED before flush begins; popup reads state from chrome.storage.session (synchronously updated), not from in-memory variable.
#13 smart-sync.ps1 path separator on non-WindowsHardcoded backslash path separator broke sync when run under PowerShell Core on macOS/Linux. Fix: all path construction uses Join-Path for OS-agnostic separator resolution.
